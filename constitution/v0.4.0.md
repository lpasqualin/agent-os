# Agent OS — Architecture Constitution v0.4.0

**March 18, 2026 — FROZEN**

> *The chassis owns the mounting points. The parts are swappable. The contracts are sacred.*

This document is the architectural constitution of the Agent OS. It defines invariants, contracts, and guarantees. It does not prescribe implementation mechanisms. It is only reopened if implementation exposes a structural problem.

---

## 1. Core Principles

**Principle 1: Own the contracts, not the engines.** The Agent OS defines interfaces between layers. It does not implement memory, observability, or model inference. Those are pluggable backends.

**Principle 2: Every layer is swappable while operational.** Changing a memory backend, observability provider, or runtime must not require rewriting agents. Adapters absorb the difference.

**Principle 3: Agents are portable artifacts.** An agent definition is a declarative spec that can run on any compliant runtime. No runtime-specific logic lives inside an agent definition.

**Principle 4: Governance is not optional.** Every side-effecting action passes through automatic policy evaluation. Human approval is required only for gated actions. These are distinct mechanisms (§7).

**Principle 5: Observe everything, surface nothing by default.** All runs, tool calls, memory operations, and failures are traced. Traces are structured and queryable. They never leak into user-facing channels unless explicitly configured.

**Principle 6: No premature scaling.** Build for one operator with one box first. Multi-tenancy, distributed execution, and cloud deployment are future concerns.

**Principle 7: Agents define intent, runs are the operational unit.** The agent spec declares what an agent is. The run is what the system manages: traces, governance, cost, retries, and approvals all happen per run.

**Principle 8: Define guarantees, not mechanisms.** The constitution specifies what must be true, never how it must be achieved. Adapter implementations choose their own mechanisms as long as guarantees are met.

> ⚠ **HARD LAW:** No side-effecting action executes without governance policy evaluation first. No exceptions. This means automatic, cheap policy checks on every action — not human approval on every action. Human interruption is reserved for gated action classes only.

## 3. Action Taxonomy

Every action belongs to one of these classes. This defines "side effect" precisely for governance.

| Action Class | Definition | Governance Default | Audit Default |
|-------------|-----------|-------------------|--------------|
| pure_read | No mutation, no cost, no privacy concern | Allow (no eval needed) | Minimal |
| sensitive_read | Privacy-sensitive or crosses trust boundary | Allow with audit | Full |
| billable_read | Incurs cost per call | Allow with budget check | Standard |
| internal_mutation | Changes state within agent's own scope | Allow | Standard |
| external_mutation | Changes state in an external system | Confirm for unvetted; allow for established | Full |
| irreversible_mutation | Cannot be undone | Confirm | Full with justification |
| privileged_control | Changes to the agent system itself | Confirm | Full with justification |

**Rules:**
- Every capability must declare its action class
- Governance defaults derive from action class; agent specs can override stricter (never more permissive)
- When in doubt, classify up
- Action class is stable across runtimes

> ⚠ **HARD LAW:** pure_read is the only action class exempt from governance evaluation. All other classes require at minimum automatic policy evaluation before execution.

## 4. The Agent Contract

### 4.1 Capability Model

Agent specs declare abstract capabilities using `domain.verb` naming. Runtime adapters map capabilities to concrete implementations.

#### 4.1.1 Capability Naming Doctrine

| Rule | Description | Example |
|------|-----------|---------|
| Format | Always `domain.verb` — two segments, dot-separated | `tasks.read`, `calendar.list` |
| Domain | Resource being acted upon. Singular noun. | tasks, calendar, web, browser, email, memory, files |
| Verb | Action being performed. Present tense. | read, write, list, search, create, delete, navigate, extract |
| Granularity | One capability = one atomic permission boundary | `browser.navigate` ≠ `browser.extract` |
| No compound verbs | Never combine multiple actions | `read_and_write` is wrong |
| No runtime specifics | Never reference a tool name or brand | `todoist.read` is wrong — use `tasks.read` |

#### 4.1.2 Capability Semantic Contract

Every registered capability carries these guarantees:

| Guarantee | What It Specifies | Why It Matters |
|----------|------------------|---------------|
| action_class | Which action taxonomy class (§3) | Governance defaults, audit level |
| idempotent | Repeating with same input = same result, no extra side effects | Retry safety |
| compensable | Can be undone or reversed | Rollback capability |
| data_sensitivity | none / internal / personal / regulated / financial | Audit verbosity |
| justification_required | Agent must provide reasoning before executing | Required for irreversible_mutation and privileged_control |

## 5. Run Lifecycle

> ✎ **DESIGN NOTE:** Agent spec defines intent. Run is the operational unit. Implementation should be run-centric.

| State | Description | Transitions To |
|-------|-----------|---------------|
| created | Initialized, not yet triggered | scheduled, planning, canceled |
| scheduled | Queued for future execution | planning, canceled |
| planning | Agent reasoning about next action(s) | awaiting_approval, executing, failed |
| awaiting_approval | Governance paused for human confirmation | executing, canceled, failed |
| executing | Running tool calls / generating output | planning, succeeded, failed |
| succeeded | Completed successfully | (terminal) |
| failed | Terminated due to error (see §5.1) | (terminal) or retrying |
| retrying | Policy allows retry | planning, failed |
| canceled | Operator/governance canceled | (terminal) |

**Key rules:**
- Every state transition is traced
- Policy evaluation is automatic and cheap (milliseconds)
- Human approval only for `confirm` capabilities
- **All runs pass through planning** — even trivial tasks. Planning may be instantaneous but is never skipped.
- Approval timeout: 15 min default, configurable
- Canceled always available for non-terminal runs

### 5.1 Failure Taxonomy

| Failure Reason | Description | Default Retry | Alert Level |
|---------------|-----------|--------------|------------|
| policy_denied | Governance denied the action | No | Warning |
| approval_timeout | Human approval not received in time | No | Info |
| budget_exceeded | Hit spend limit | No (until reset) | Warning |
| tool_failed | Tool/skill returned error | Retryable if idempotent | Error |
| backend_unavailable | Memory/obs/runtime unreachable | Retryable with backoff | Critical |
| model_failed | LLM call failed | Retryable + fallback | Warning |
| validation_failed | Invalid output or constraint violation | No | Error |
| operator_canceled | Explicit cancellation | No | Info |

> ✓ **GUARANTEE:** Every failed run must carry a specific failure reason from this taxonomy. Adapters must not collapse distinct failures into generic errors.

## 6. Adapter Contracts

### 6.1 Runtime Adapter

> ✎ **DESIGN NOTE:** If boundary becomes strained, split into Resolver + RuntimeController. Watch this seam.

- `deploy(agent_spec, env_binding)` → Resolve config, map capabilities, deploy
- `start(agent_id)` → Start agent
- `stop(agent_id)` → Stop gracefully
- `status(agent_id)` → Health, run state, metadata
- `list()` → All agents with state
- `execute(agent_id, task)` → One-shot task, returns run_id
- `resolve_capability(capability_id)` → Runtime-native tool for capability

#### 6.1.1 Runtime Compliance Levels

| Level | Name | What It Means |
|-------|------|--------------|
| L1 | Spec-compatible | Can parse and deploy agent.yaml. Capabilities mapped. Agent starts. No semantic equivalence guarantee. |
| L2 | Semantically-compatible | Preserves capability behavior. Memory, governance, observability contracts fully honored. |
| L3 | Operationally-verified | Passed test suite validating behavior, governance, failure handling, audit, performance. |

### 6.2 Memory Adapter

- `remember(agent_id, entry)` → Store per schema (§6.2.1)
- `recall(agent_id, query, filters)` → Semantic search
- `forget(agent_id, selector)` → Remove by ID/category/filter
- `list(agent_id, filters)` → Structured query (no semantic matching)
- `compact(agent_id)` → Summarize, merge, prune
- `export(agent_id, format)` → JSON/YAML export (required for portability)

#### 6.2.1 Memory Entry Schema

| Field | Type | Required | Description |
|-------|------|---------|------------|
| id | string | auto | Generated on remember() |
| category | enum | yes | fact / preference / event / decision / learning / context |
| content | string | yes | The memory content |
| scope | enum | yes | agent / shared / global |
| confidence | float | no | 0.0–1.0, default 1.0 |
| source | enum | no | user_stated / inferred / tool_output / system |
| supersedes | string | no | ID of entry this replaces |
| created_at | timestamp | auto | Creation time |
| expires_at | timestamp | no | Optional TTL |

#### 6.2.2 Memory Category Laws

| Category | What Qualifies | Durability | Rules |
|---------|---------------|-----------|-------|
| fact | Verifiable statement | Durable | Must have source. Supersedes conflicts. |
| preference | User-expressed or repeated pattern | Durable | Never from single observation. |
| event | Something at a specific time | 30-day TTL | Must include timestamp. Promote only if significant. |
| decision | Choice with rationale | Durable | Must include reasoning. Never deleted, only superseded. |
| learning | Insight from past runs | Durable | Created after runs. Reference source run_id. |
| context | Temporary working context | 24-hour TTL | Strict TTL. Never promoted. Compaction prunes aggressively. |

> ⚠ **HARD LAW:** tool_output must never become a memory entry without summarization. Raw output is traced via observability. Memory entries must be extracted facts, decisions, or learnings.

#### 6.2.3 Memory Scope Authority

| Scope | Who Can Write | Who Can Read | Conflict Resolution | Governance |
|-------|--------------|-------------|-------------------|-----------|
| agent | Only owning agent | Only owning agent | N/A | internal_mutation |
| shared | Agents with shared_write privilege | Agents with shared_read privilege | Last-write-wins with supersedes chain | external_mutation; first write to new namespace requires confirm |
| global | Only agents with global_write (highest trust) | All agents | Operator arbitrates; conflicts trigger approval | privileged_control — always confirm |

> ✓ **GUARANTEE:** No agent may write to shared or global scope without appropriate privilege. Promotion from agent → shared/global requires a new remember() at target scope — no silent promotion.

### 6.3 Observability Adapter

- `trace_start(run_id, agent_id, metadata)` → Begin trace
- `trace_event(run_id, event_type, payload)` → Record: tool_call, memory_op, llm_call, state_transition, governance_decision, error
- `trace_end(run_id, status, failure_reason, metadata)` → Complete trace
- `query(filters)` → Query traces
- `dashboard_url(agent_id)` → Link to backend dashboard

### 6.4 Governance Adapter

| Mechanism | When | Latency | Blocking? |
|----------|------|---------|----------|
| Policy evaluation | Every side effect (except pure_read) | Milliseconds | Non-blocking for 'allow' |
| Human approval | Only for policy = 'confirm' | Minutes | Blocks run |

- `evaluate(agent_id, action, context)` → allow / deny / require_approval (must be fast)
- `request_approval(agent_id, action, context)` → approved / denied / timeout
- `audit_log(agent_id, run_id, action, result, metadata)` → Append-only
- `get_policy(agent_id)` → Active policy
- `set_policy(agent_id, policy)` → Update policy
- `check_budget(agent_id)` → Remaining budget

#### 6.4.1 Audit Trail Guarantees

> ✓ **GUARANTEE:** Audit records must be ordered, tamper-evident, and independently verifiable outside the runtime. Implementation may use any mechanism — but modification or deletion after creation must be detectable.

- **Ordered:** Monotonically increasing sequence per agent
- **Tamper-evident:** Post-creation modification detectable
- **Exportable:** Complete trail exportable as structured data
- **Independently verifiable:** Integrity verifiable without trusting runtime/adapter

## 7. Governance Model

### 7.1 Two-Tier Enforcement

**Tier 1: Automatic policy evaluation.** Every side-effecting action. In-process, millisecond latency. Checks: capability allowed? Budget? Rate limit? Action class default? Result: allow / deny / escalate.

**Tier 2: Human approval gate.** Only when Tier 1 returns require_approval. Sends request via operator channel. Blocks until response or timeout.

### 7.2 Policy Rules

- Per-capability access control from action class defaults (overridable stricter only)
- Per-agent spend limits: tokens/cost per hour, day, month
- Rate limits: tool calls per minute, concurrent executions
- Data sensitivity enforcement: regulated/financial → elevated audit
- Escalation rules for human deferral

### 7.3 Approval Gates

- All irreversible_mutation defaults to confirm
- All privileged_control defaults to confirm
- external_mutation to unvetted targets defaults to confirm on first use
- Defaults overridable per agent (stricter only)
- Timeout: 15 min default, configurable

## 8. Portability Rules

1. No agent logic references specific runtime/memory/obs platform. All through adapters.
2. Agent specs are source of truth. Runtime state is derived.
3. Capabilities use `domain.verb`, not runtime-native IDs.
4. Memory exportable as structured data. No proprietary formats.
5. Traces conform to OpenTelemetry-compatible schemas where possible.
6. Chassis testable with mock adapters for all layers.
7. Secrets resolved from secure storage at deploy time only.
8. Config in three layers: portable spec, env binding, resolved deployment. Only first two persisted.
9. Portability = spec portability AND semantic portability. Adapters declare compliance level (§6.1.1).

## 11. What We Do NOT Build

- **A memory engine.** We build MemoryAdapter, not a vector database.
- **An observability platform.** We build ObsAdapter, not a tracing backend.
- **A new LLM runtime.** We build RuntimeAdapter, not an execution engine.
- **A workflow framework.** We adopt if needed.
- **A model router.** Runtime's job.
- **A channel system.** Runtime connects; we declare bindings.
- **A tool marketplace.** We declare capabilities; adapters map them.
